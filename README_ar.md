# ai-initializer

**مُولّد سياق المشروع التلقائي لأدوات البرمجة بالذكاء الاصطناعي**

> يفحص مجلد مشروعك ويُنشئ تلقائيًا
> `AGENTS.md` + ملفات سياق المعرفة/المهارات/الأدوار حتى تتمكن عوامل الذكاء الاصطناعي من بدء العمل فورًا.

```
أمر واحد → تحليل المشروع → إنشاء AGENTS.md → يعمل مع أي أداة ذكاء اصطناعي
```

---

## الاستخدام

> **ملاحظة حول استهلاك الرموز** — أثناء الإعداد الأولي، يقوم نموذج متقدم بتحليل المشروع بالكامل وإنشاء ملفات متعددة (AGENTS.md، .ai-agents/context/، .ai-agents/skills/، .ai-agents/roles/). قد يستهلك هذا عشرات الآلاف من الرموز حسب حجم المشروع. هذه تكلفة لمرة واحدة؛ الجلسات اللاحقة تُحمّل السياق المُنشأ مسبقًا وتبدأ فورًا.

```bash
# 1. اجعل الذكاء الاصطناعي يقرأ HOW_TO_AGENTS.md وسيتولى الباقي

# الخيار أ: الإنجليزية (مُوصى به — استهلاك أقل للرموز، أداء أمثل للذكاء الاصطناعي)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "Read HOW_TO_AGENTS.md and generate AGENTS.md tailored to this project"

# الخيار ب: لغة المستخدم (مُوصى به إذا كنت تخطط لتحرير AGENTS.md يدويًا)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "HOW_TO_AGENTS.md를 읽고 이 프로젝트에 맞게 AGENTS.md를 생성하라"

# مُوصى به: --model claude-opus-4-6 (أو أحدث) للحصول على أفضل النتائج
# مُوصى به: --dangerously-skip-permissions للتنفيذ المستقل دون انقطاع

# 2. ابدأ العمل مع العوامل المُنشأة
./ai-agency.sh
```

## لماذا تحتاج هذا؟

أدوات البرمجة بالذكاء الاصطناعي **تُعيد تعلّم المشروع من الصفر** في كل جلسة.

| المشكلة | النتيجة |
|---|---|
| لا تعرف اتفاقيات الفريق | تناقضات في أسلوب الكود |
| لا تعرف خريطة API الكاملة | تستكشف قاعدة الكود بالكامل في كل مرة (تكلفة +20%) |
| لا تعرف الإجراءات المحظورة | عمليات خطرة مثل الوصول المباشر لقاعدة بيانات الإنتاج |
| لا تعرف تبعيات الخدمات | تأثيرات جانبية مفقودة |

**ai-initializer** يحل هذه المشكلة — أنشئ مرة واحدة، وأي أداة ذكاء اصطناعي ستفهم مشروعك فورًا.

## كيف يعمل

### الخطوة 1: فحص المشروع وتصنيفه

يستكشف المجلدات حتى عمق 3 ويُصنّف تلقائيًا حسب أنماط الملفات.

```
deployment.yaml + service.yaml  →  k8s-workload
values.yaml (Helm)              →  infra-component
package.json + *.tsx            →  frontend
go.mod                          →  backend-go
Dockerfile + CI config          →  cicd
...19 نوعًا يتم اكتشافها تلقائيًا
```

### الخطوة 2: إنشاء السياق

يُنشئ ملفات المعرفة في `.ai-agents/context/` عن طريق **تحليل الكود فعليًا** بناءً على الأنواع المكتشفة.

```
تم اكتشاف خدمة خلفية
  → فحص المسارات/المتحكمات → إنشاء api-spec.json
  → فحص الكيانات/المخططات  → إنشاء data-model.md
  → فحص إعدادات Kafka      → إنشاء event-spec.json
```

### الخطوة 3: إنشاء AGENTS.md

يُنشئ AGENTS.md لكل مجلد باستخدام القوالب المناسبة.

```
AGENTS.md الجذر (الاتفاقيات العامة)
  → الالتزامات: Conventional Commits
  → PR: قالب مطلوب، موافقة واحدة على الأقل
  → الفروع: feature/{ticket}-{desc}
       │
       ▼ وراثة تلقائية (لا تُكرر في الأبناء)
  apps/api/AGENTS.md
    → تجاوزات فقط: "هذه الخدمة تستخدم Python"
```

### الخطوة 4: التهيئة الخاصة بالمزود

يُضيف جسور إلى إعدادات المزود المحددة حتى **تقرأ جميع أدوات الذكاء الاصطناعي** ملف AGENTS.md المُنشأ.

```
┌──────────────┐     ┌─────────────┐     ┌─────────────┐
│ Claude Code  │     │   Cursor    │     │   Codex     │
│  CLAUDE.md   │     │  .mdc rules │     │  AGENTS.md  │
│      ↓       │     │      ↓      │     │  (أصلي)     │
│ "اقرأ        │     │ "اقرأ       │     │      ✓      │
│  AGENTS.md"  │     │  AGENTS.md" │     │             │
└──────┬───────┘     └──────┬──────┘     └─────────────┘
       └──────────┬─────────┘
                  ▼
           AGENTS.md (مصدر الحقيقة الوحيد)
                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
  .ai-agents/  .ai-agents/  .ai-agents/
   context/     skills/      roles/
```

> **المبدأ:** ملفات التهيئة تُنشأ فقط للمزودين المستخدمين فعليًا. لا يتم إنشاء ملفات إعدادات للأدوات غير المستخدمة.

## هيكل العوامل الهرمي

```
┌───────────────────────────────────────┐
│  عامل مدير المشروع الجذر (AGENTS.md) │
│  الاتفاقيات العامة + قواعد التفويض   │
│  "التحقق من التصميم > التحقق من الكود"│
└────────┬──────────┬─────────┬────────┘
         │          │         │
    ┌────▼────┐ ┌───▼────┐ ┌──▼─────┐
    │ خبير   │ │ البنية │ │ مخطط  │
    │ الخدمة │ │التحتية │ │الوثائق│
    └─────────┘ └────────┘ └────────┘

التفويض: الأب → الابن (يعمل ضمن نطاق AGENTS.md لذلك المجلد)
التقرير: الابن → الأب (ملخص التغييرات بعد إتمام المهمة)
التنسيق: لا تنسيق مباشر بين الأقران — تنسيق غير مباشر عبر الأب
```

## تحسين الرموز

| الصيغة | عدد الرموز | ملاحظات |
|---|---|---|
| وصف API بلغة طبيعية | ~200 رمز | |
| JSON DSL | ~70 رمزًا | **توفير 3 أضعاف** |

**مثال api-spec.json:**
```json
{
  "service": "order-api",
  "apis": [{
    "method": "POST",
    "path": "/api/v1/orders",
    "domains": ["Order", "Payment"],
    "sideEffects": ["kafka:order-created", "db:orders.insert"]
  }]
}
```

**هدف AGENTS.md:** أقل من **300 رمز** بعد الاستبدال

## بروتوكول استعادة الجلسة

```
بدء الجلسة:
  1. قراءة AGENTS.md (معظم أدوات الذكاء الاصطناعي تفعل ذلك تلقائيًا)
  2. اتباع مسارات ملفات السياق لتحميل .ai-agents/context/
  3. فحص .ai-agents/context/current-work.md (العمل الجاري)
  4. git log --oneline -10 (فهم التغييرات الأخيرة)

نهاية الجلسة:
  1. العمل الجاري → تسجيله في current-work.md
  2. معرفة المجال المكتسبة حديثًا → تحديث ملفات السياق
  3. المهام غير المكتملة → تسجيلها صراحةً
```

## صيانة السياق

عند تغيير الكود، يجب تحديث ملفات `.ai-agents/context/` وفقًا لذلك.

```
إضافة/تغيير/حذف API        →  تحديث api-spec.json
تغيير مخطط قاعدة البيانات   →  تحديث data-model.md
تغيير مواصفات الأحداث       →  تحديث event-spec.json
تغيير سياسة الأعمال         →  تحديث domain-overview.md
تغيير التكامل الخارجي       →  تحديث external-integration.md
تغيير إعدادات البنية التحتية →  تحديث infra-spec.md
```

> عدم التحديث يعني أن الجلسة التالية ستعمل **بسياق قديم**.
