# ai-initializer

**Автоматический генератор контекста проекта для AI-инструментов разработки**

> Сканирует директорию вашего проекта и автоматически генерирует
> `AGENTS.md` + контекст знаний/навыков/ролей, чтобы AI-агенты могли начать работу мгновенно.

```
Одна команда → Анализ проекта → Генерация AGENTS.md → Работает с любым AI-инструментом
```

---

## Использование

> **Примечание о расходе токенов** — При первоначальной настройке модель высшего уровня анализирует весь проект и генерирует несколько файлов (AGENTS.md, .ai-agents/context/, .ai-agents/skills/, .ai-agents/roles/). Это может потребовать десятки тысяч токенов в зависимости от размера проекта. Это единоразовые затраты; последующие сессии загружают заранее созданный контекст и запускаются мгновенно.

```bash
# 1. Дайте AI прочитать HOW_TO_AGENTS.md — он сделает всё остальное

# Вариант A: На английском (рекомендуется — меньше расход токенов, оптимальная производительность AI)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "Read HOW_TO_AGENTS.md and generate AGENTS.md tailored to this project"

# Вариант B: На языке пользователя (рекомендуется, если вы планируете вручную редактировать AGENTS.md)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "HOW_TO_AGENTS.md를 읽고 이 프로젝트에 맞게 AGENTS.md를 생성하라"

# Рекомендуется: --model claude-opus-4-6 (или новее) для лучших результатов
# Рекомендуется: --dangerously-skip-permissions для непрерывного автономного выполнения

# 2. Начните работу с созданными агентами
./ai-agency.sh
```

## Зачем это нужно?

AI-инструменты разработки **заново изучают проект с нуля** каждую сессию.

| Проблема | Последствие |
|---|---|
| Не знает командных соглашений | Несогласованность стиля кода |
| Не знает полной карты API | Исследует всю кодовую базу каждый раз (стоимость +20%) |
| Не знает запрещённых действий | Рискованные операции, такие как прямой доступ к production-базе данных |
| Не знает зависимостей сервисов | Пропущенные побочные эффекты |

**ai-initializer** решает эту проблему — сгенерируйте один раз, и любой AI-инструмент мгновенно поймёт ваш проект.

## Основные принципы

> ETH Zurich (2026.03): **Включение выводимого содержимого снижает успешность и увеличивает стоимость на +20%**

```
Включать (невыводимое)         .ai-agents/context/ (дорогой вывод)      Исключать (дешёвый вывод)
───────────────────────        ────────────────────────────────────      ────────────────────────
Командные соглашения           Полная карта API                          Структура директорий
Запрещённые действия           Связи модели данных                       Содержимое отдельных файлов
Форматы PR/коммитов            Спецификации pub/sub событий              Официальная документация фреймворков
Скрытые зависимости            Топология инфраструктуры                  Связи импортов
```

## Генерируемая структура

```
project-root/
├── AGENTS.md                          # PM-агент (общая оркестрация)
├── .ai-agents/
│   ├── context/                       # Файлы знаний (загружаются при старте сессии)
│   │   ├── domain-overview.md         #   Бизнес-домен, политики, ограничения
│   │   ├── data-model.md              #   Определения сущностей, связи, переходы состояний
│   │   ├── api-spec.json              #   Карта API (JSON DSL, экономия токенов в 3 раза)
│   │   ├── event-spec.json            #   Спецификации событий Kafka/MQ
│   │   ├── infra-spec.md              #   Helm-чарты, сеть, порядок деплоя
│   │   └── external-integration.md    #   Внешние API, аутентификация, лимиты запросов
│   ├── skills/                        # Поведенческие процессы (загружаются по запросу)
│   │   ├── develop/SKILL.md           #   Разработка: анализ → проектирование → реализация → тестирование → PR
│   │   ├── deploy/SKILL.md            #   Деплой: тег → запрос на деплой → проверка
│   │   ├── review/SKILL.md            #   Ревью: по чек-листу
│   │   ├── hotfix/SKILL.md            #   Процесс экстренного исправления
│   │   └── context-update/SKILL.md    #   Процедура обновления файлов контекста
│   └── roles/                         # Определения ролей (глубина контекста по ролям)
│       ├── pm.md                      #   Менеджер проекта
│       ├── backend.md                 #   Backend-разработчик
│       ├── frontend.md                #   Frontend-разработчик
│       ├── sre.md                     #   SRE / Инфраструктура
│       └── reviewer.md                #   Ревьюер кода
│
├── apps/
│   ├── api/AGENTS.md                  # Агенты для каждого сервиса
│   └── web/AGENTS.md
└── infra/
    └── helm/AGENTS.md
```

## Как это работает

### Шаг 1: Сканирование и классификация проекта

Исследует директории до глубины 3 и автоматически классифицирует по шаблонам файлов.

```
deployment.yaml + service.yaml  →  k8s-workload
values.yaml (Helm)              →  infra-component
package.json + *.tsx            →  frontend
go.mod                          →  backend-go
Dockerfile + CI config          →  cicd
...19 типов определяются автоматически
```

### Шаг 2: Генерация контекста

Генерирует файлы знаний `.ai-agents/context/`, **фактически анализируя код** на основе обнаруженных типов.

```
Обнаружен backend-сервис
  → Сканирование маршрутов/контроллеров → Генерация api-spec.json
  → Сканирование сущностей/схем         → Генерация data-model.md
  → Сканирование конфигурации Kafka      → Генерация event-spec.json
```

### Шаг 3: Генерация AGENTS.md

Генерирует AGENTS.md для каждой директории с использованием соответствующих шаблонов.

```
Корневой AGENTS.md (Глобальные соглашения)
  → Коммиты: Conventional Commits
  → PR: Шаблон обязателен, 1+ одобрение
  → Ветки: feature/{ticket}-{desc}
       │
       ▼ Автоматически наследуется (не дублируется в дочерних)
  apps/api/AGENTS.md
    → Переопределяет только: "Этот сервис использует Python"
```

### Шаг 4: Вендор-специфичная инициализация

Добавляет мосты к вендор-специфичным конфигурациям, чтобы **все AI-инструменты читали** сгенерированный AGENTS.md.

```
┌──────────────┐     ┌─────────────┐     ┌─────────────┐
│ Claude Code  │     │   Cursor    │     │   Codex     │
│  CLAUDE.md   │     │  .mdc rules │     │  AGENTS.md  │
│      ↓       │     │      ↓      │     │  (нативно)  │
│ "прочитать   │     │ "прочитать  │     │      ✓      │
│  AGENTS.md"  │     │  AGENTS.md" │     │             │
└──────┬───────┘     └──────┬──────┘     └─────────────┘
       └──────────┬─────────┘
                  ▼
           AGENTS.md (единый источник истины)
                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
  .ai-agents/  .ai-agents/  .ai-agents/
   context/     skills/      roles/
```

> **Принцип:** Файлы инициализации генерируются только для уже используемых вендоров. Конфигурационные файлы для неиспользуемых инструментов никогда не создаются.

## Совместимость с инструментами

| Инструмент | Автоматически читает AGENTS.md | Инициализация |
|---|---|---|
| **OpenAI Codex** | Да (нативно) | Не требуется |
| **Claude Code** | Частично (резервный вариант) | Добавляет директиву в `CLAUDE.md` |
| **Cursor** | Нет | Добавляет `.mdc` в `.cursor/rules/` |
| **GitHub Copilot** | Нет | Генерирует `.github/copilot-instructions.md` |
| **Windsurf** | Нет | Добавляет директиву в `.windsurfrules` |
| **Aider** | Да | Добавляет чтение в `.aider.conf.yml` |

Автоматическая генерация инициализации:
```bash
bash scripts/sync-ai-rules.sh
```

## Иерархическая структура агентов

```
┌───────────────────────────────────────┐
│  Корневой PM-агент (AGENTS.md)        │
│  Глобальные соглашения + Правила      │
│  делегирования                        │
│  "Валидация дизайна > Валидация кода" │
└────────┬──────────┬─────────┬────────┘
         │          │         │
    ┌────▼────┐ ┌───▼────┐ ┌──▼─────┐
    │ Эксперт │ │ Инфра  │ │Планиро-│
    │ сервиса │ │  SRE   │ │вщик док│
    └─────────┘ └────────┘ └────────┘

Делегирование: Родитель → Дочерний (работает в рамках AGENTS.md своей директории)
Отчётность:    Дочерний → Родитель (сводка изменений после завершения задачи)
Координация:   Нет прямого взаимодействия между равными — косвенная координация через родителя
```

## Оптимизация токенов

| Формат | Количество токенов | Примечания |
|---|---|---|
| Описание API на естественном языке | ~200 токенов | |
| JSON DSL | ~70 токенов | **Экономия в 3 раза** |

**Пример api-spec.json:**
```json
{
  "service": "order-api",
  "apis": [{
    "method": "POST",
    "path": "/api/v1/orders",
    "domains": ["Order", "Payment"],
    "sideEffects": ["kafka:order-created", "db:orders.insert"]
  }]
}
```

**Целевой размер AGENTS.md:** Менее **300 токенов** после подстановки

## Протокол восстановления сессии

```
Начало сессии:
  1. Прочитать AGENTS.md (большинство AI-инструментов делают это автоматически)
  2. Следовать по путям файлов контекста для загрузки .ai-agents/context/
  3. Проверить .ai-agents/context/current-work.md (текущая работа)
  4. git log --oneline -10 (понять последние изменения)

Конец сессии:
  1. Текущая работа → Записать в current-work.md
  2. Вновь полученные знания о домене → Обновить файлы контекста
  3. Незавершённые TODO → Записать явно
```

## Поддержка контекста

При изменении кода файлы `.ai-agents/context/` должны быть обновлены соответственно.

```
API добавлен/изменён/удалён          →  Обновить api-spec.json
Схема БД изменена                    →  Обновить data-model.md
Спецификация событий изменена        →  Обновить event-spec.json
Бизнес-политика изменена             →  Обновить domain-overview.md
Внешняя интеграция изменена          →  Обновить external-integration.md
Конфигурация инфраструктуры изменена →  Обновить infra-spec.md
```

> Если не обновлять, следующая сессия будет **работать с устаревшим контекстом**.
