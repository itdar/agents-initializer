# ai-initializer

**Автоматический генератор контекста проекта для AI-инструментов программирования**

> Сканирует директорию вашего проекта и автоматически генерирует
> `AGENTS.md` + контекст знаний/навыков/ролей, чтобы AI-агенты могли начать работу немедленно.

```
Одна команда → Анализ проекта → Генерация AGENTS.md → Работает с любым AI-инструментом
```

---

## Использование

> **Уведомление о расходе токенов** — Во время начальной настройки модель высшего уровня анализирует весь проект и генерирует несколько файлов (AGENTS.md, .ai-agents/context/, .ai-agents/skills/, .ai-agents/roles/). Это может потребовать десятки тысяч токенов в зависимости от размера проекта. Это разовая стоимость; последующие сессии загружают предварительно созданный контекст и запускаются мгновенно.

```bash
# 1. Дайте AI прочитать HOW_TO_AGENTS.md — остальное он сделает сам

# Вариант A: Английский (рекомендуется — меньше расход токенов, оптимальная производительность AI)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "Read HOW_TO_AGENTS.md and generate AGENTS.md tailored to this project"

# Вариант B: Язык пользователя (рекомендуется, если вы планируете вручную редактировать AGENTS.md)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "HOW_TO_AGENTS.md를 읽고 이 프로젝트에 맞게 AGENTS.md를 생성하라"

# Рекомендуется: --model claude-opus-4-6 (или новее) для лучших результатов
# Рекомендуется: --dangerously-skip-permissions для непрерывного автономного выполнения

# 2. Начните работать с созданными агентами
./ai-agency.sh
```

## Зачем это нужно?

### Проблема: AI теряет память после каждой сессии

```
 Сессия 1                  Сессия 2                  Сессия 3
┌──────────┐             ┌──────────┐             ┌──────────┐
│ AI читает │             │ AI читает │             │ Начинаем  │
│ весь      │  Сессия     │ весь      │  Сессия     │ с нуля    │
│ код       │  завершена  │ код       │  завершена  │ снова     │
│ (30 мин)  │ ──────→    │ (30 мин)  │ ──────→    │ (30 мин)  │
│ Начинает  │ Память     │ Начинает  │ Память     │ Начинает  │
│ работать  │ потеряна!  │ работать  │ потеряна!  │ работать  │
│           │             │           │             │           │
└──────────┘             └──────────┘             └──────────┘
```

AI-агенты забывают всё при завершении сессии. Каждый раз они тратят время на понимание структуры проекта, анализ API и изучение соглашений.

| Проблема | Последствие |
|---|---|
| Не знает соглашений команды | Несоответствие стиля кода |
| Не знает полной карты API | Каждый раз исследует всю кодовую базу (стоимость +20%) |
| Не знает запрещённых действий | Рискованные операции, например прямой доступ к продакшн-базе |
| Не знает зависимостей сервисов | Пропущенные побочные эффекты |

### Решение: Предварительно создать «мозг» для AI

```
 Начало сессии
┌──────────────────────────────────────────────────┐
│                                                  │
│  Читает AGENTS.md (автоматически)                │
│       │                                          │
│       ▼                                          │
│  "Я — эксперт по бэкенду этого сервиса"         │
│  "Соглашения: Conventional Commits, TypeScript   │
│   strict"                                        │
│  "Запрещено: модификация других сервисов,        │
│   хардкод секретов"                              │
│       │                                          │
│       ▼                                          │
│  Загружает файлы .ai-agents/context/ (5 секунд)  │
│  "20 API, 15 сущностей, 8 событий понятны"       │
│       │                                          │
│       ▼                                          │
│  Немедленно приступает к работе!                 │
│                                                  │
└──────────────────────────────────────────────────┘
```

**ai-initializer** решает эту проблему — сгенерируйте один раз, и любой AI-инструмент мгновенно поймёт ваш проект.

## Основной принцип: 3-уровневая архитектура

```
                    Ваш проект
                         │
            ┌────────────┼────────────┐
            ▼            ▼            ▼

     ┌──────────┐  ┌──────────┐  ┌──────────┐
     │ AGENTS.md│  │.ai-agents│  │.ai-agents│
     │          │  │ /context/ │  │ /skills/ │
     │Идентич-  │  │  Знания  │  │Поведение │
     │  ность   │  │ "Что я   │  │ "Как я   │
     │ "Кто я?" │  │  знаю?"  │  │ работаю?"│
     │          │  │          │  │          │
     │ + Правила│  │ + Домен  │  │ + Деплой │
     │ + Доступы│  │ + Модели │  │ + Ревью  │
     └──────────┘  └──────────┘  └──────────┘
      Точка входа  Хранилище      Стандарты
                   памяти         рабочих процессов
```

### 1. AGENTS.md — «Кто я?»

Файл **идентичности** агента, развёрнутого в каждой директории.

```
project/
├── AGENTS.md                  ← PM: Лидер, координирующий всё
├── apps/
│   └── api/
│       └── AGENTS.md          ← Эксперт API: Отвечает только за этот сервис
├── infra/
│   ├── AGENTS.md              ← SRE: Управляет всей инфраструктурой
│   └── monitoring/
│       └── AGENTS.md          ← Специалист по мониторингу
└── configs/
    └── AGENTS.md              ← Менеджер конфигурации
```

Это работает как **организационная структура команды**:
- PM контролирует всё и распределяет задачи
- Каждый участник команды глубоко разбирается только в своей области
- Они не выполняют работу других команд напрямую — они запрашивают её

### 2. `.ai-agents/context/` — «Что я знаю?»

Папка, в которой **необходимые знания предварительно систематизированы**, чтобы AI не приходилось читать код каждый раз.

```
.ai-agents/context/
├── domain-overview.md     ← "Этот сервис управляет заказами..."
├── data-model.md          ← "Есть сущности Order, Payment, Delivery..."
├── api-spec.json          ← "POST /orders, GET /orders/{id}, ..."
└── event-spec.json        ← "Публикует событие order-created..."
```

**Аналогия:** Документация для онбординга нового сотрудника. Задокументируйте один раз — и никому не придётся объяснять снова.

### 3. `.ai-agents/skills/` — «Как я работаю?»

**Стандартизированные руководства** для повторяющихся задач.

```
.ai-agents/skills/
├── develop/SKILL.md       ← "Разработка: Анализ → Проектирование → Реализация → Тесты → PR"
├── deploy/SKILL.md        ← "Деплой: Тег → Запрос → Проверка"
└── review/SKILL.md        ← "Ревью: Безопасность, Производительность, Чеклист тестов"
```

**Аналогия:** Операционное руководство команды — заставляет AI следовать правилам, таким как «проверь этот чеклист перед отправкой PR».

## Что писать, а что нет

> ETH Zurich (2026.03): **Включение выводимого содержимого снижает показатели успешности и увеличивает стоимость на +20%**

```
         Писать это                       Не писать это
  ┌─────────────────────────┐     ┌─────────────────────────┐
  │                         │     │                         │
  │  "Используйте feat:    │     │  "Исходный код в        │
  │   для коммитов"         │     │   папке src/"           │
  │  AI не может это        │     │  AI видит это через ls  │
  │   вывести               │     │                         │
  │                         │     │                         │
  │  "Нет прямого пуша      │     │  "React основан на      │
  │   в main"               │     │   компонентах"          │
  │  Правило команды,       │     │  Уже в официальной      │
  │   не в коде             │     │   документации          │
  │                         │     │                         │
  │  "Перед деплоем         │     │  "Этот файл содержит    │
  │   требуется одобрение   │     │   100 строк"            │
  │   QA-команды"           │     │  AI может прочитать     │
  │  Процесс, невыводимый   │     │   это напрямую          │
  │                         │     │                         │
  └─────────────────────────┘     └─────────────────────────┘
       Писать в AGENTS.md             НЕ писать!
```

**Исключение:** «То, что можно вывести, но делать это каждый раз слишком дорого»

```
  напр.: Полный список API (нужно прочитать 20 файлов, чтобы разобраться)
  напр.: Связи модели данных (разбросаны по 10 файлам)
  напр.: Межсервисные вызовы (нужно проверить и код, и инфраструктуру)

  → Предварительно систематизируйте это в .ai-agents/context/!
  → В AGENTS.md указывайте только путь: «ищи здесь»
```

```
Включать (невыводимое)          .ai-agents/context/ (дорогой вывод)       Исключать (дешёвый вывод)
──────────────────────          ──────────────────────────────────        ───────────────────────
Соглашения команды              Полная карта API                          Структура директорий
Запрещённые действия            Связи модели данных                       Содержимое отдельных файлов
Форматы PR/коммитов             Спецификации событий pub/sub              Официальная документация фреймворков
Скрытые зависимости             Топология инфраструктуры                  Связи импортов
```

## Как это работает

### Шаг 1: Сканирование и классификация проекта

Исследует директории на глубину до 3 и автоматически классифицирует по шаблонам файлов.

```
deployment.yaml + service.yaml  →  k8s-workload
values.yaml (Helm)              →  infra-component
package.json + *.tsx            →  frontend
go.mod                          →  backend-go
Dockerfile + CI config          →  cicd
...19 типов определяются автоматически
```

### Шаг 2: Генерация контекста

Генерирует файлы знаний `.ai-agents/context/`, **фактически анализируя код** на основе обнаруженных типов.

```
Обнаружен бэкенд-сервис
  → Сканирование routes/controllers → Генерация api-spec.json
  → Сканирование entities/schemas   → Генерация data-model.md
  → Сканирование Kafka config       → Генерация event-spec.json
```

### Шаг 3: Генерация AGENTS.md

Генерирует AGENTS.md для каждой директории, используя соответствующие шаблоны.

```
Корневой AGENTS.md (Глобальные соглашения)
  → Коммиты: Conventional Commits
  → PR: Шаблон обязателен, 1+ одобрение
  → Ветки: feature/{ticket}-{desc}
       │
       ▼ Автоматическое наследование (не повторяется в дочерних)
  apps/api/AGENTS.md
    → Переопределяет только: "Этот сервис использует Python"
```

Глобальные правила используют **паттерн наследования** — написали в одном месте, и оно автоматически применяется далее по иерархии.

```
Корневой AGENTS.md ──────────────────────────────────────────
│ Глобальные соглашения:
│  - Коммиты: Conventional Commits (feat:, fix:, chore:)
│  - PR: Шаблон обязателен, минимум 1 ревьюер
│  - Ветки: feature/{ticket}-{desc}
│
│     Автонаследование              Автонаследование
│     ┌──────────────────┐       ┌──────────────────┐
│     ▼                  │       ▼                  │
│  apps/api/AGENTS.md    │    infra/AGENTS.md       │
│  (Указываются только   │    (Указываются только   │
│   дополнительные       │     дополнительные       │
│   правила)             │     правила)             │
│  "Этот сервис          │    "При изменении Helm   │
│   использует Python"   │     values — сначала     │
│                        │     спроси"              │
└─────────────────────────┴──────────────────────────
```

**Преимущества:**
- Хотите изменить правила коммитов? → Измените только в корне
- Добавляете новый сервис? → Глобальные правила применяются автоматически
- Нужны особые правила для конкретного сервиса? → Переопределите в его AGENTS.md

### Шаг 4: Vendor-специфичная начальная настройка

Добавляет мосты к vendor-специфичным конфигурациям, чтобы **все AI-инструменты читали** сгенерированный AGENTS.md.

```
┌──────────────┐     ┌─────────────┐     ┌─────────────┐
│ Claude Code  │     │   Cursor    │     │   Codex     │
│  CLAUDE.md   │     │  .mdc rules │     │  AGENTS.md  │
│      ↓       │     │      ↓      │     │  (нативно)  │
│ "читай       │     │ "читай      │     │      ✓      │
│  AGENTS.md"  │     │  AGENTS.md" │     │             │
└──────┬───────┘     └──────┬──────┘     └─────────────┘
       └──────────┬─────────┘
                  ▼
           AGENTS.md (единый источник истины)
                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
  .ai-agents/  .ai-agents/  .ai-agents/
   context/     skills/      roles/
```

> **Принцип:** Файлы начальной настройки генерируются только для уже используемых инструментов. Конфигурационные файлы для неиспользуемых инструментов никогда не создаются.

## Сгенерированная структура

```
project-root/
├── AGENTS.md                          # PM-агент (общая оркестрация)
├── .ai-agents/
│   ├── context/                       # Файлы знаний (загружаются при старте сессии)
│   │   ├── domain-overview.md         #   Бизнес-домен, политики, ограничения

│   │   ├── event-spec.json            #   Спецификации событий Kafka/MQ
│   │   ├── infra-spec.md              #   Helm-чарты, сети, порядок деплоя
│   │   └── external-integration.md    #   Внешние API, аутентификация, лимиты запросов
│   ├── skills/                        # Поведенческие рабочие процессы (загружаются по запросу)
│   │   ├── develop/SKILL.md           #   Разработка: анализ → проектирование → реализация → тесты → PR
│   │   ├── deploy/SKILL.md            #   Деплой: тег → запрос на деплой → проверка
│   │   ├── review/SKILL.md            #   Ревью: по чеклисту
│   │   ├── hotfix/SKILL.md            #   Рабочий процесс экстренного исправления
│   │   └── context-update/SKILL.md    #   Процедура обновления контекстных файлов
│   └── roles/                         # Определения ролей (глубина контекста по ролям)
│       ├── pm.md                      #   Менеджер проекта
│       ├── backend.md                 #   Бэкенд-разработчик
│       ├── frontend.md                #   Фронтенд-разработчик
│       ├── sre.md                     #   SRE / Инфраструктура
│       └── reviewer.md               #   Код-ревьюер
│
├── apps/
│   ├── api/AGENTS.md                  # Агенты для каждого сервиса
│   └── web/AGENTS.md
└── infra/
    └── helm/AGENTS.md
```

## Запуск сессий

После настройки всех агентов выберите нужного агента и сразу запустите сессию.

```bash
$ ./ai-agency.sh

=== AI Agent Sessions ===
Найдено: 8 агент(ов)

  1) [PM] project-root
  2) api-service
  3) monitoring
  ...

Выберите агента (номер): 2

=== AI Tool ===
  1) claude
  2) codex
  3) print

Выберите инструмент: 1

→ Сессия запущена в директории api-service
→ Агент автоматически загружает AGENTS.md и .ai-agents/context/
→ Готов к работе немедленно!
```

**Параллельное выполнение (tmux):**

```bash
$ ./ai-agency.sh --multi

Выберите агентов: 1,2,3   # Запустить PM + API + Monitoring одновременно

→ Открыто 3 tmux-сессии
→ Разные агенты работают независимо в каждой панели
→ Переключение панелей через Ctrl+B N
```

## Оптимизация токенов

| Формат | Количество токенов | Примечания |
|---|---|---|
| Описание API на естественном языке | ~200 токенов | |
| JSON DSL | ~70 токенов | **экономия в 3 раза** |

**Пример api-spec.json:**
```json
{
  "service": "order-api",
  "apis": [{
    "method": "POST",
    "path": "/api/v1/orders",
    "domains": ["Order", "Payment"],
    "sideEffects": ["kafka:order-created", "db:orders.insert"]
  }]
}
```

**Цель для AGENTS.md:** Менее **300 токенов** после подстановки

## Протокол восстановления сессии

```
Начало сессии:
  1. Прочитать AGENTS.md (большинство AI-инструментов делают это автоматически)
  2. Следовать путям контекстных файлов для загрузки .ai-agents/context/
  3. Проверить .ai-agents/context/current-work.md (незавершённая работа)
  4. git log --oneline -10 (понять недавние изменения)

Завершение сессии:
  1. Незавершённая работа → Записать в current-work.md
  2. Новые знания о домене → Обновить контекстные файлы
  3. Невыполненные TODO → Записать явно
```

## Поддержание контекста

При изменении кода файлы `.ai-agents/context/` должны обновляться соответственно.

```
API добавлен/изменён/удалён     →  Обновить api-spec.json
Схема БД изменена               →  Обновить data-model.md
Спецификация событий изменена   →  Обновить event-spec.json
Бизнес-политика изменена        →  Обновить domain-overview.md
Внешняя интеграция изменена     →  Обновить external-integration.md
Конфигурация инфраструктуры     →  Обновить infra-spec.md
  изменена
```

> Отсутствие обновлений означает, что следующая сессия будет **работать с устаревшим контекстом**.

## Аналогия: Традиционная команда vs Команда AI-агентов

```
              Традиционная команда       Команда AI-агентов
              ────────────────────       ──────────────────
 Лидер        PM (человек)               Корневой AGENTS.md (PM-агент)
 Участники    N разработчиков            AGENTS.md в каждой директории
 Онбординг    Confluence/Notion          .ai-agents/context/
 Руководства  Вики команды               .ai-agents/skills/
 Роли         Должности/R&R документы    .ai-agents/roles/
 Правила      Соглашения команды         Global Conventions (наследуемые)
 Приход       Приходит в офис            Сессия начинается → AGENTS.md загружен
 Уход         Уходит (память сохраняется) Сессия завершается (память потеряна!)
 След. день   Память на месте            .ai-agents/context/ загружен (память восстановлена)
```

**Ключевое отличие:** Люди сохраняют память после ухода с работы, но AI забывает всё каждый раз.
Именно поэтому существует `.ai-agents/context/` — он служит **долговременной памятью** AI.

## Артефакты

| Файл | Аудитория | Назначение |
|---|---|---|
| `HOW_TO_AGENTS.md` | AI | Мета-инструкция, которую агенты читают и выполняют |
| `README.md` | Человек | Этот документ — руководство для понимания человеком |
| `ai-agency.sh` | Человек | Выбор агента → Запуск AI-сессии |
| `AGENTS.md` (каждая директория) | AI | Идентичность агента + правила для директории |
| `.ai-agents/context/*.md/json` | AI | Предварительно систематизированные знания о домене |
| `.ai-agents/skills/*/SKILL.md` | AI | Стандартизированные рабочие процессы |
| `.ai-agents/roles/*.md` | AI/Человек | Стратегии загрузки контекста по ролям |
