# ai-initializer

**AI कोडिंग टूल्स के लिए स्वचालित प्रोजेक्ट संदर्भ जनरेटर**

> आपकी प्रोजेक्ट डायरेक्टरी को स्कैन करता है और स्वचालित रूप से
> `AGENTS.md` + ज्ञान/कौशल/भूमिका संदर्भ उत्पन्न करता है ताकि AI एजेंट तुरंत काम शुरू कर सकें।

```
One command → Project analysis → AGENTS.md generation → Works with any AI tool
```

---

## उपयोग

> **टोकन उपयोग सूचना** — प्रारंभिक सेटअप के दौरान, एक शीर्ष-स्तरीय मॉडल पूरे प्रोजेक्ट का विश्लेषण करता है और कई फाइलें उत्पन्न करता है (AGENTS.md, .ai-agents/context/, .ai-agents/skills/, .ai-agents/roles/)। प्रोजेक्ट के आकार के आधार पर इसमें हजारों टोकन खर्च हो सकते हैं। यह एक बार का खर्च है; बाद के सत्र पूर्व-निर्मित संदर्भ लोड करते हैं और तुरंत शुरू होते हैं।

```bash
# 1. AI को HOW_TO_AGENTS.md पढ़ने दें और बाकी वह संभाल लेगा

# विकल्प A: अंग्रेज़ी (अनुशंसित — कम टोकन लागत, इष्टतम AI प्रदर्शन)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "Read HOW_TO_AGENTS.md and generate AGENTS.md tailored to this project"

# विकल्प B: उपयोगकर्ता की भाषा (अनुशंसित अगर आप AGENTS.md को मैन्युअली संपादित करने की योजना बनाते हैं)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "HOW_TO_AGENTS.md पढ़ो और इस प्रोजेक्ट के अनुसार AGENTS.md बनाओ"

# अनुशंसित: --model claude-opus-4-6 (या बाद का) सर्वोत्तम परिणामों के लिए
# अनुशंसित: --dangerously-skip-permissions निर्बाध स्वायत्त निष्पादन के लिए

# 2. उत्पन्न एजेंट्स के साथ काम शुरू करें
./ai-agency.sh
```

| समस्या | परिणाम |

### 1. AGENTS.md — "मैं कौन हूं?"

प्रत्येक डायरेक्टरी में तैनात एजेंट की **पहचान फाइल**।

```
project/
├── AGENTS.md                  ← PM: सब कुछ समन्वयित करने वाला नेता
├── apps/
│   └── api/
│       └── AGENTS.md          ← API विशेषज्ञ: केवल इस सेवा के लिए जिम्मेदार
├── infra/
│   ├── AGENTS.md              ← SRE: सभी बुनियादी ढांचे का प्रबंधन
│   └── monitoring/
│       └── AGENTS.md          ← मॉनिटरिंग विशेषज्ञ
└── configs/
    └── AGENTS.md              ← कॉन्फ़िगरेशन प्रबंधक
```

यह बिल्कुल **टीम संगठन चार्ट** की तरह काम करता है:
- PM सब कुछ देखरेख करता है और कार्य वितरित करता है
- प्रत्येक टीम सदस्य केवल अपने क्षेत्र को गहराई से समझता है
- वे सीधे अन्य टीमों का काम नहीं करते — वे अनुरोध करते हैं

### 2. `.ai-agents/context/` — "मुझे क्या पता है?"

एक फोल्डर जहां **आवश्यक ज्ञान पहले से व्यवस्थित** किया जाता है ताकि AI को हर बार कोड पढ़ना न पड़े।

```
.ai-agents/context/
├── domain-overview.md     ← "यह सेवा ऑर्डर प्रबंधन संभालती है..."
├── data-model.md          ← "Order, Payment, Delivery एंटिटीज़ हैं..."
├── api-spec.json          ← "POST /orders, GET /orders/{id}, ..."
└── event-spec.json        ← "order-created इवेंट प्रकाशित करता है..."
```

**उपमा:** नए कर्मचारी के लिए ऑनबोर्डिंग दस्तावेज़ीकरण। एक बार दस्तावेज़ बनाएं, और किसी को दोबारा समझाना नहीं पड़ेगा।

### 3. `.ai-agents/skills/` — "मैं कैसे काम करता हूं?"

दोहराए जाने वाले कार्यों के लिए **मानकीकृत कार्यप्रवाह मैनुअल**।

```
.ai-agents/skills/
├── develop/SKILL.md       ← "फीचर विकास: विश्लेषण → डिज़ाइन → कार्यान्वयन → परीक्षण → PR"
├── deploy/SKILL.md        ← "डिप्लॉय: टैग → अनुरोध → सत्यापन"
└── review/SKILL.md        ← "रिव्यू: सुरक्षा, प्रदर्शन, परीक्षण चेकलिस्ट"
```

**उपमा:** टीम का संचालन मैनुअल — AI को "PR सबमिट करने से पहले यह चेकलिस्ट जांचें" जैसे नियमों का पालन कराता है।

## यह कैसे काम करता है

### चरण 1: प्रोजेक्ट स्कैन और वर्गीकरण

गहराई 3 तक डायरेक्टरी की खोज करता है और फाइल पैटर्न के आधार पर स्वचालित वर्गीकरण करता है।

```
deployment.yaml + service.yaml  →  k8s-workload
values.yaml (Helm)              →  infra-component
package.json + *.tsx            →  frontend
go.mod                          →  backend-go
Dockerfile + CI config          →  cicd
...19 प्रकार स्वचालित पहचाने जाते हैं
```

### चरण 2: संदर्भ उत्पन्न करना

पता लगाए गए प्रकारों के आधार पर **वास्तव में कोड का विश्लेषण करके** `.ai-agents/context/` ज्ञान फाइलें उत्पन्न करता है।

```
बैकएंड सेवा का पता चला
  → routes/controllers स्कैन करें → api-spec.json उत्पन्न करें
  → entities/schemas स्कैन करें   → data-model.md उत्पन्न करें
  → Kafka config स्कैन करें       → event-spec.json उत्पन्न करें
```

### चरण 3: AGENTS.md उत्पन्न करना

उपयुक्त टेम्पलेट्स का उपयोग करके प्रत्येक डायरेक्टरी के लिए AGENTS.md उत्पन्न करता है।

```
रूट AGENTS.md (वैश्विक परंपराएं)
  → कमिट्स: Conventional Commits
  → PR: टेम्पलेट आवश्यक, 1+ स्वीकृतियां
  → ब्रांच: feature/{ticket}-{desc}
       │
       ▼ स्वचालित विरासत (चाइल्ड में दोहराया नहीं जाता)
  apps/api/AGENTS.md
    → केवल ओवरराइड: "यह सेवा Python उपयोग करती है"
```

वैश्विक नियम **विरासत पैटर्न** का उपयोग करते हैं — एक जगह लिखें, और यह स्वचालित रूप से नीचे लागू होता है।

```
रूट AGENTS.md ──────────────────────────────────────────
│ वैश्विक परंपराएं:
│  - कमिट्स: Conventional Commits (feat:, fix:, chore:)
│  - PR: टेम्पलेट आवश्यक, कम से कम 1 समीक्षक
│  - ब्रांच: feature/{ticket}-{desc}
│
│     स्वचालित विरासत               स्वचालित विरासत
│     ┌──────────────────┐       ┌──────────────────┐
│     ▼                  │       ▼                  │
│  apps/api/AGENTS.md    │    infra/AGENTS.md       │
│  (केवल अतिरिक्त       │    (केवल अतिरिक्त       │
│   नियम निर्दिष्ट)      │     नियम निर्दिष्ट)      │
│  "यह सेवा Python      │    "Helm values बदलते    │
│   उपयोग करती है"       │     समय, पहले पूछें"     │
└─────────────────────────┴──────────────────────────
```

**लाभ:**
- कमिट नियम बदलना चाहते हैं? → केवल रूट में संशोधित करें
- नई सेवा जोड़ रहे हैं? → वैश्विक नियम स्वचालित रूप से लागू होते हैं
- किसी विशिष्ट सेवा के लिए अलग नियम चाहिए? → उस सेवा के AGENTS.md में ओवरराइड करें

### चरण 4: विक्रेता-विशिष्ट बूटस्ट्रैप

विक्रेता-विशिष्ट कॉन्फ़िग में ब्रिज जोड़ता है ताकि **सभी AI टूल** उत्पन्न AGENTS.md को पढ़ सकें।

```
┌──────────────┐     ┌─────────────┐     ┌─────────────┐
│ Claude Code  │     │   Cursor    │     │   Codex     │
│  CLAUDE.md   │     │  .mdc rules │     │  AGENTS.md  │
│      ↓       │     │      ↓      │     │  (native)   │
│ "read        │     │ "read       │     │      ✓      │
│  AGENTS.md"  │     │  AGENTS.md" │     │             │
└──────┬───────┘     └──────┬──────┘     └─────────────┘
       └──────────┬─────────┘
                  ▼
           AGENTS.md (एकल सत्य स्रोत)
                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
  .ai-agents/  .ai-agents/  .ai-agents/
   context/     skills/      roles/
```

> **सिद्धांत:** बूटस्ट्रैप फाइलें केवल उन विक्रेताओं के लिए उत्पन्न होती हैं जो पहले से उपयोग में हैं। अप्रयुक्त टूल्स के लिए कॉन्फ़िग फाइलें कभी नहीं बनाई जातीं।

## सत्र लॉन्चर

सभी एजेंट सेट होने के बाद, वांछित एजेंट चुनें और तुरंत सत्र शुरू करें।

```bash
$ ./ai-agency.sh

=== AI एजेंट सत्र ===
पाए गए: 8 एजेंट

  1) [PM] project-root
  2) api-service
  3) monitoring
  ...

एजेंट चुनें (नंबर): 2

=== AI टूल ===
  1) claude
  2) codex
  3) print

टूल चुनें: 1

→ api-service डायरेक्टरी में सत्र शुरू हुआ
→ एजेंट स्वचालित रूप से AGENTS.md और .ai-agents/context/ लोड करता है
→ तुरंत काम के लिए तैयार!
```

**समानांतर निष्पादन (tmux):**

```bash
$ ./ai-agency.sh --multi

एजेंट चुनें: 1,2,3   # PM + API + Monitoring एक साथ चलाएं

→ 3 tmux सत्र खुले
→ प्रत्येक पैन में अलग-अलग एजेंट स्वतंत्र रूप से काम करते हैं
→ Ctrl+B N से पैन बदलें
```

## संदर्भ रखरखाव

जब कोड बदलता है, तो `.ai-agents/context/` फाइलों को तदनुसार अपडेट करना होगा।

```
API जोड़ा/बदला/हटाया गया      →  api-spec.json अपडेट करें
DB स्कीमा बदला                 →  data-model.md अपडेट करें
इवेंट विनिर्देश बदला            →  event-spec.json अपडेट करें
व्यावसायिक नीति बदली            →  domain-overview.md अपडेट करें
बाहरी एकीकरण बदला              →  external-integration.md अपडेट करें
बुनियादी ढांचा कॉन्फ़िग बदला    →  infra-spec.md अपडेट करें
```

> अपडेट न करने का मतलब है कि अगला सत्र **पुराने संदर्भ के साथ काम करेगा**।
