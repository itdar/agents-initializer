# ai-initializer

**مُولِّد سياق المشروع التلقائي لأدوات البرمجة بالذكاء الاصطناعي**

> يفحص مجلد مشروعك ويُنشئ تلقائياً
> `AGENTS.md` + سياق المعرفة/المهارات/الأدوار حتى يتمكن وكلاء الذكاء الاصطناعي من البدء بالعمل فوراً.

```
أمر واحد → تحليل المشروع → إنشاء AGENTS.md → يعمل مع أي أداة ذكاء اصطناعي
```

---

## الاستخدام

> **ملاحظة حول استهلاك التوكنات** — أثناء الإعداد الأولي، يقوم نموذج متقدم بتحليل المشروع بالكامل وإنشاء ملفات متعددة (AGENTS.md، .ai-agents/context/، .ai-agents/skills/، .ai-agents/roles/). قد يستهلك هذا عشرات الآلاف من التوكنات حسب حجم المشروع. هذه تكلفة لمرة واحدة؛ الجلسات اللاحقة تُحمّل السياق المُعَدّ مسبقاً وتبدأ فوراً.

```bash
# 1. اجعل الذكاء الاصطناعي يقرأ HOW_TO_AGENTS.md وسيتولى الباقي

# الخيار أ: الإنجليزية (موصى به — استهلاك أقل للتوكنات، أداء مثالي للذكاء الاصطناعي)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "Read HOW_TO_AGENTS.md and generate AGENTS.md tailored to this project"

# الخيار ب: بلغة المستخدم (موصى به إذا كنت تخطط لتعديل AGENTS.md يدوياً)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "اقرأ HOW_TO_AGENTS.md وأنشئ AGENTS.md مخصص لهذا المشروع"

# موصى به: --model claude-opus-4-6 (أو أحدث) للحصول على أفضل النتائج
# موصى به: --dangerously-skip-permissions للتنفيذ المستقل دون انقطاع

# 2. ابدأ العمل مع الوكلاء المُنشأين
./ai-agency.sh
```

### المشكلة: الذكاء الاصطناعي يفقد ذاكرته كل جلسة

```
 الجلسة 1                   الجلسة 2                   الجلسة 3
┌──────────┐             ┌──────────┐             ┌──────────┐
│ الذكاء    │             │ الذكاء    │             │ البدء     │
│ الاصطناعي │  انتهاء     │ الاصطناعي │  انتهاء     │ من        │
│ يقرأ كامل │  الجلسة     │ يقرأ كامل │  الجلسة     │ الصفر     │
│ الكود     │ ──────→    │ الكود     │ ──────→    │ مجدداً    │
│ (30 دقيقة)│ فُقدت      │ (30 دقيقة)│ فُقدت      │ (30 دقيقة)│
│ يبدأ      │ الذاكرة!   │ يبدأ      │ الذاكرة!   │ يبدأ      │
│ العمل     │             │ العمل     │             │ العمل     │
└──────────┘             └──────────┘             └──────────┘
```

وكلاء الذكاء الاصطناعي ينسون كل شيء عند انتهاء الجلسة. في كل مرة، يقضون وقتاً في فهم بنية المشروع وتحليل الواجهات البرمجية وتعلم الاتفاقيات.

| المشكلة | النتيجة |
|---|---|
| لا يعرف اتفاقيات الفريق | تناقضات في أسلوب الكود |
| لا يعرف خريطة الواجهات البرمجية الكاملة | يستكشف كامل الكود في كل مرة (تكلفة +20%) |
| لا يعرف الإجراءات المحظورة | عمليات خطرة مثل الوصول المباشر لقاعدة بيانات الإنتاج |
| لا يعرف تبعيات الخدمات | تأثيرات جانبية غير محسوبة |

## المبدأ الأساسي: بنية من 3 طبقات

```
                    مشروعك
                         │
            ┌────────────┼────────────┐
            ▼            ▼            ▼

     ┌──────────┐  ┌──────────┐  ┌──────────┐
     │ AGENTS.md│  │.ai-agents│  │.ai-agents│
     │          │  │ /context/ │  │ /skills/ │
     │ الهوية   │  │ المعرفة  │  │ السلوك   │
     │ "من      │  │ "ماذا    │  │ "كيف     │
     │  أنا؟"   │  │  أعرف؟"  │  │  أعمل؟"  │
     │          │  │          │  │          │
     │ + قواعد  │  │ + المجال │  │ + النشر  │
     │ + صلاحيات│  │ + نماذج  │  │ + مراجعة │
     │ + مسارات │  │          │  │          │
     └──────────┘  └──────────┘  └──────────┘
      نقطة الدخول   مخزن الذاكرة  معايير سير العمل
```

### 1. AGENTS.md — "من أنا؟"

**ملف الهوية** للوكيل المُوزَّع في كل مجلد.

```
project/
├── AGENTS.md                  ← مدير المشروع: القائد الذي ينسق كل شيء
├── apps/
│   └── api/
│       └── AGENTS.md          ← خبير الواجهات البرمجية: مسؤول عن هذه الخدمة فقط
├── infra/
│   ├── AGENTS.md              ← SRE: يدير كل البنية التحتية
│   └── monitoring/
│       └── AGENTS.md          ← متخصص المراقبة
└── configs/
    └── AGENTS.md              ← مدير التكوينات
```

يعمل تماماً مثل **الهيكل التنظيمي للفريق**:
- مدير المشروع يُشرف على كل شيء ويوزع المهام
- كل عضو في الفريق يفهم بعمق مجاله فقط
- لا يتعاملون مباشرة مع عمل الفرق الأخرى — بل يطلبونه

### 2. `.ai-agents/context/` — "ماذا أعرف؟"

مجلد يتم فيه **تنظيم المعرفة الأساسية مسبقاً** حتى لا يضطر الذكاء الاصطناعي لقراءة الكود في كل مرة.

```
.ai-agents/context/
├── domain-overview.md     ← "هذه الخدمة تتعامل مع إدارة الطلبات..."
├── data-model.md          ← "يوجد كيانات Order, Payment, Delivery..."
├── api-spec.json          ← "POST /orders, GET /orders/{id}, ..."
└── event-spec.json        ← "تنشر حدث order-created..."
```

**تشبيه:** وثائق التأهيل للموظف الجديد. وثّقها مرة واحدة، ولن يضطر أحد لشرحها مجدداً.

### 3. `.ai-agents/skills/` — "كيف أعمل؟"

**أدلة سير عمل موحدة** للمهام المتكررة.

```
.ai-agents/skills/
├── develop/SKILL.md       ← "تطوير الميزات: تحليل → تصميم → تنفيذ → اختبار → PR"
├── deploy/SKILL.md        ← "النشر: وسم → طلب → تحقق"
└── review/SKILL.md        ← "المراجعة: قائمة تحقق الأمان والأداء والاختبارات"
```

**تشبيه:** دليل عمليات الفريق — يجعل الذكاء الاصطناعي يتبع قواعد مثل "تحقق من هذه القائمة قبل تقديم PR."

## ماذا تكتب وماذا لا تكتب

> ETH Zurich (2026.03): **تضمين محتوى يمكن استنتاجه يُقلل معدلات النجاح ويزيد التكلفة بنسبة +20%**

```
         اكتب هذا                          لا تكتب هذا
  ┌─────────────────────────┐     ┌─────────────────────────┐
  │                         │     │                         │
  │  "استخدم صيغة feat:     │     │  "الكود المصدري في      │
  │   للإيداعات"            │     │   مجلد src/"            │
  │  الذكاء الاصطناعي لا    │     │  يمكنه رؤية هذا بـ ls   │
  │  يستطيع استنتاج هذا     │     │                         │
  │                         │     │  "React يعتمد على       │
  │  "ممنوع الدفع المباشر   │     │   المكونات"             │
  │   إلى main"            │     │  موجود في الوثائق       │
  │  قاعدة فريق، ليست في   │     │   الرسمية               │
  │  الكود                  │     │                         │
  │                         │     │  "هذا الملف طوله 100    │
  │  "موافقة فريق QA        │     │   سطر"                  │
  │   مطلوبة قبل النشر"     │     │  يمكن للذكاء الاصطناعي  │
  │  عملية، لا يمكن         │     │   قراءته مباشرة         │
  │  استنتاجها              │     │                         │
  │                         │     │                         │
  └─────────────────────────┘     └─────────────────────────┘
       اكتبها في AGENTS.md              لا تكتبها!
```

**استثناء:** "أشياء يمكن استنتاجها لكن تكلفة استنتاجها عالية جداً في كل مرة"

```
  مثال: قائمة الواجهات البرمجية الكاملة (يجب قراءة 20 ملفاً لمعرفتها)
  مثال: علاقات نموذج البيانات (موزعة عبر 10 ملفات)
  مثال: علاقات الاستدعاء بين الخدمات (يجب فحص الكود + البنية التحتية)

  → نظّم هذه مسبقاً في .ai-agents/context/!
  → في AGENTS.md، اكتب فقط المسار: "اذهب هنا لتجده"
```

```

───────────────────────        ────────────────────────────────────      ────────────────────────

الإجراءات المحظورة              علاقات نموذج البيانات                     محتويات ملف واحد
صيغ PR/الإيداعات                مواصفات نشر/اشتراك الأحداث               وثائق الإطار الرسمية
التبعيات المخفية                طوبولوجيا البنية التحتية                 علاقات الاستيراد
```

## كيف يعمل

### الخطوة 1: فحص المشروع وتصنيفه

يستكشف المجلدات حتى العمق 3 ويصنّف تلقائياً حسب أنماط الملفات.

```
deployment.yaml + service.yaml  →  k8s-workload
values.yaml (Helm)              →  infra-component
package.json + *.tsx            →  frontend
go.mod                          →  backend-go
Dockerfile + CI config          →  cicd
...19 نوعاً يتم اكتشافها تلقائياً
```

### الخطوة 2: إنشاء السياق

يُنشئ ملفات المعرفة في `.ai-agents/context/` عبر **تحليل الكود فعلياً** بناءً على الأنواع المكتشفة.

```
تم اكتشاف خدمة خلفية
  → فحص routes/controllers → إنشاء api-spec.json
  → فحص entities/schemas   → إنشاء data-model.md
  → فحص إعدادات Kafka      → إنشاء event-spec.json
```

### الخطوة 3: إنشاء AGENTS.md

يُنشئ AGENTS.md لكل مجلد باستخدام القوالب المناسبة.

```
AGENTS.md الجذر (الاتفاقيات العامة)
  → الإيداعات: Conventional Commits
  → PR: قالب مطلوب، موافقة واحدة على الأقل
  → الفروع: feature/{ticket}-{desc}
       │
       ▼ وراثة تلقائية (لا تتكرر في الأبناء)
  apps/api/AGENTS.md
    → تجاوزات فقط: "هذه الخدمة تستخدم Python"
```

القواعد العامة تستخدم **نمط الوراثة** — اكتب في مكان واحد، وتُطبَّق تلقائياً في الأسفل.

```
AGENTS.md الجذر ──────────────────────────────────────────
│ الاتفاقيات العامة:
│  - الإيداعات: Conventional Commits (feat:, fix:, chore:)
│  - PR: قالب مطلوب، مراجع واحد على الأقل
│  - الفروع: feature/{ticket}-{desc}
│
│     وراثة تلقائية              وراثة تلقائية
│     ┌──────────────────┐       ┌──────────────────┐
│     ▼                  │       ▼                  │
│  apps/api/AGENTS.md    │    infra/AGENTS.md       │
│  (فقط القواعد          │    (فقط القواعد          │
│   الإضافية)            │     الإضافية)            │
│  "هذه الخدمة تستخدم    │    "عند تغيير قيم Helm   │
│   Python"              │     اسأل أولاً"          │
└─────────────────────────┴──────────────────────────
```

**الفوائد:**
- تريد تغيير قواعد الإيداع؟ → عدّل الجذر فقط
- إضافة خدمة جديدة؟ → القواعد العامة تُطبَّق تلقائياً
- تحتاج قواعد مختلفة لخدمة معينة؟ → تجاوزها في AGENTS.md الخاص بتلك الخدمة

### الخطوة 4: ربط الأدوات المختلفة

يُضيف جسوراً لإعدادات الأدوات المختلفة حتى **تقرأ جميع أدوات الذكاء الاصطناعي** ملف AGENTS.md المُنشأ.

```
┌──────────────┐     ┌─────────────┐     ┌─────────────┐
│ Claude Code  │     │   Cursor    │     │   Codex     │
│  CLAUDE.md   │     │  .mdc rules │     │  AGENTS.md  │
│      ↓       │     │      ↓      │     │  (أصلي)     │
│ "اقرأ        │     │ "اقرأ       │     │      ✓      │
│  AGENTS.md"  │     │  AGENTS.md" │     │             │
└──────┬───────┘     └──────┬──────┘     └─────────────┘
       └──────────┬─────────┘
                  ▼
           AGENTS.md (مصدر الحقيقة الوحيد)
                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
  .ai-agents/  .ai-agents/  .ai-agents/
   context/     skills/      roles/
```

> **المبدأ:** ملفات الربط تُنشأ فقط للأدوات المستخدمة بالفعل. لا يتم إنشاء ملفات تكوين لأدوات غير مستخدمة.

## مُشغّل الجلسات

بعد إعداد جميع الوكلاء، اختر الوكيل المطلوب وابدأ جلسة فوراً.

```bash
$ ./ai-agency.sh

=== جلسات وكلاء الذكاء الاصطناعي ===
تم العثور على: 8 وكيل/وكلاء

  1) [PM] project-root
  2) api-service
  3) monitoring
  ...

اختر الوكيل (رقم): 2

=== أداة الذكاء الاصطناعي ===
  1) claude
  2) codex
  3) print

اختر الأداة: 1

→ بدأت الجلسة في مجلد api-service
→ الوكيل يُحمّل تلقائياً AGENTS.md و .ai-agents/context/
→ جاهز للعمل فوراً!
```

**التنفيذ المتوازي (tmux):**

```bash
$ ./ai-agency.sh --multi

اختر الوكلاء: 1,2,3   # تشغيل PM + API + Monitoring في وقت واحد

→ 3 جلسات tmux مفتوحة
→ وكلاء مختلفون يعملون بشكل مستقل في كل لوحة
→ التبديل بين اللوحات بـ Ctrl+B N
```

| وصف الواجهة البرمجية بلغة طبيعية | ~200 توكن | |
| JSON DSL | ~70 توكن | **توفير 3 أضعاف** |

**مثال api-spec.json:**
```json
{
  "service": "order-api",
  "apis": [{
    "method": "POST",
    "path": "/api/v1/orders",
    "domains": ["Order", "Payment"],
    "sideEffects": ["kafka:order-created", "db:orders.insert"]
  }]
}
```

**هدف AGENTS.md:** أقل من **300 توكن** بعد الاستبدال

## بروتوكول استعادة الجلسة

```
بداية الجلسة:
  1. قراءة AGENTS.md (معظم أدوات الذكاء الاصطناعي تفعل هذا تلقائياً)
  2. اتباع مسارات ملفات السياق لتحميل .ai-agents/context/
  3. فحص .ai-agents/context/current-work.md (العمل الجاري)
  4. git log --oneline -10 (فهم التغييرات الأخيرة)

نهاية الجلسة:
  1. العمل الجاري → تسجيله في current-work.md
  2. المعرفة المجالية المكتسبة حديثاً → تحديث ملفات السياق
  3. المهام غير المكتملة → تسجيلها بشكل صريح
```

## صيانة السياق

عند تغيير الكود، يجب تحديث ملفات `.ai-agents/context/` وفقاً لذلك.

```
إضافة/تغيير/حذف واجهة برمجية  →  تحديث api-spec.json
تغيير مخطط قاعدة البيانات       →  تحديث data-model.md
تغيير مواصفات الأحداث           →  تحديث event-spec.json
تغيير سياسة الأعمال             →  تحديث domain-overview.md
تغيير التكامل الخارجي           →  تحديث external-integration.md
تغيير إعدادات البنية التحتية    →  تحديث infra-spec.md
```

> عدم التحديث يعني أن الجلسة التالية ستعمل **بسياق قديم**.

## تشبيه: الفريق التقليدي مقابل فريق وكلاء الذكاء الاصطناعي

```
              فريق التطوير التقليدي      فريق وكلاء الذكاء الاصطناعي
              ────────────────────       ──────────────────
 القائد       مدير مشروع (إنسان)         AGENTS.md الجذر (وكيل PM)
 الأعضاء      N مطورين                  AGENTS.md في كل مجلد
 التأهيل      Confluence/Notion          .ai-agents/context/
 الأدلة       ويكي الفريق               .ai-agents/skills/
 تعريف الأدوار عناوين وظيفية/مستندات R&R .ai-agents/roles/
 قواعد الفريق  مستندات اتفاقيات الفريق   الاتفاقيات العامة (موروثة)
 بدء العمل    الوصول للمكتب              الجلسة تبدأ → AGENTS.md يُحمَّل
 انتهاء العمل المغادرة (الذاكرة محفوظة)  الجلسة تنتهي (الذاكرة تُفقد!)
 اليوم التالي  الذاكرة سليمة              .ai-agents/context/ يُحمَّل (الذاكرة تُستعاد)
```

**الفرق الجوهري:** البشر يحتفظون بذاكرتهم بعد مغادرة العمل، لكن الذكاء الاصطناعي ينسى كل شيء في كل مرة.
لهذا السبب يوجد `.ai-agents/context/` — فهو بمثابة **الذاكرة طويلة المدى** للذكاء الاصطناعي.
