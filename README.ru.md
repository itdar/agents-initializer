# ai-initializer

**Автоматический генератор контекста проекта для AI-инструментов разработки**

> Сканирует каталог вашего проекта и автоматически генерирует
> `AGENTS.md` + файлы знаний/навыков/ролей, чтобы AI-агенты могли приступить к работе немедленно.

```
Одна команда → Анализ проекта → Генерация AGENTS.md → Работает с любым AI-инструментом
```

---

## Использование

> **Обратите внимание на расход токенов** — При первоначальной настройке модель высшего уровня анализирует весь проект и генерирует несколько файлов (AGENTS.md, .ai-agents/context/, .ai-agents/skills/, .ai-agents/roles/). Это может потребовать десятки тысяч токенов в зависимости от размера проекта. Это разовая затрата; последующие сессии загружают готовый контекст и запускаются мгновенно.

```bash
# 1. Дайте AI прочитать HOW_TO_AGENTS.md — дальше он справится сам

# Вариант A: Английский (рекомендуется — меньший расход токенов, оптимальная производительность AI)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "Read HOW_TO_AGENTS.md and generate AGENTS.md tailored to this project"

# Вариант B: Язык пользователя (рекомендуется, если вы планируете редактировать AGENTS.md вручную)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "HOW_TO_AGENTS.md를 읽고 이 프로젝트에 맞게 AGENTS.md를 생성하라"

# Рекомендуется: --model claude-opus-4-6 (или новее) для лучших результатов
# Рекомендуется: --dangerously-skip-permissions для непрерывного автономного выполнения

# 2. Начните работу с сгенерированными агентами
./ai-agency.sh
```

## Зачем это нужно?

AI-инструменты разработки **изучают проект заново с нуля** каждую сессию.

| Проблема | Последствие |
|---|---|
| Не знает командные соглашения | Несоответствия в стиле кода |
| Не знает полную карту API | Исследует всю кодовую базу каждый раз (стоимость +20%) |
| Не знает запрещённые действия | Рискованные операции, например прямой доступ к продакшн-БД |
| Не знает зависимости сервисов | Упущенные побочные эффекты |

**ai-initializer** решает эту проблему — сгенерируйте один раз, и любой AI-инструмент мгновенно понимает ваш проект.

---

## Основные принципы

> ETH Zurich (2026.03): **Включение выводимого содержимого снижает успешность и увеличивает стоимость на +20%**

```
Включать (невыводимое)          .ai-agents/context/ (затратный вывод)    Исключать (дешёвый вывод)
───────────────────────         ────────────────────────────────────      ────────────────────────
Командные соглашения            Полная карта API                          Структура каталогов
Запрещённые действия            Связи модели данных                       Содержимое отдельных файлов
Форматы PR/коммитов             Спецификации событий pub/sub              Официальная документация фреймворков
Скрытые зависимости             Топология инфраструктуры                  Связи импортов
```

## Генерируемая структура

```
project-root/
├── AGENTS.md                          # PM-агент (общая оркестрация)
├── .ai-agents/
│   ├── context/                       # Файлы знаний (загружаются при старте сессии)
│   │   ├── domain-overview.md         #   Бизнес-домен, политики, ограничения
│   │   ├── data-model.md              #   Определения сущностей, связи, переходы состояний
│   │   ├── api-spec.json              #   Карта API (JSON DSL, экономия токенов в 3 раза)
│   │   ├── event-spec.json            #   Спецификации событий Kafka/MQ
│   │   ├── infra-spec.md              #   Helm-чарты, сеть, порядок развёртывания
│   │   └── external-integration.md    #   Внешние API, аутентификация, лимиты запросов
│   ├── skills/                        # Поведенческие сценарии (загружаются по требованию)
│   │   ├── develop/SKILL.md           #   Разработка: анализ → проектирование → реализация → тесты → PR
│   │   ├── deploy/SKILL.md            #   Развёртывание: тег → запрос на деплой → проверка
│   │   ├── review/SKILL.md            #   Ревью: по чек-листу
│   │   ├── hotfix/SKILL.md            #   Экстренное исправление
│   │   └── context-update/SKILL.md    #   Процедура обновления файлов контекста
│   └── roles/                         # Определения ролей (глубина контекста по роли)
│       ├── pm.md                      #   Менеджер проекта
│       ├── backend.md                 #   Бэкенд-разработчик
│       ├── frontend.md                #   Фронтенд-разработчик
│       ├── sre.md                     #   SRE / Инфраструктура
│       └── reviewer.md                #   Ревьюер кода
│
├── apps/
│   ├── api/AGENTS.md                  # Агенты отдельных сервисов
│   └── web/AGENTS.md
└── infra/
    └── helm/AGENTS.md
```

## Как это работает

### Шаг 1: Сканирование и классификация проекта

Исследует каталоги до глубины 3 и автоматически классифицирует по паттернам файлов.

```
deployment.yaml + service.yaml  →  k8s-workload
values.yaml (Helm)              →  infra-component
package.json + *.tsx            →  frontend
go.mod                          →  backend-go
Dockerfile + CI config          →  cicd
...19 типов определяются автоматически
```

### Шаг 2: Генерация контекста

Генерирует файлы знаний `.ai-agents/context/`, **фактически анализируя код** на основе определённых типов.

```
Обнаружен бэкенд-сервис
  → Сканирование маршрутов/контроллеров → Генерация api-spec.json
  → Сканирование сущностей/схем         → Генерация data-model.md
  → Сканирование конфигурации Kafka      → Генерация event-spec.json
```

### Шаг 3: Генерация AGENTS.md

Генерирует AGENTS.md для каждого каталога с использованием соответствующих шаблонов.

```
Корневой AGENTS.md (Глобальные соглашения)
  → Коммиты: Conventional Commits
  → PR: Шаблон обязателен, 1+ одобрений
  → Ветки: feature/{ticket}-{desc}
       │
       ▼ Автоматическое наследование (не повторяется в дочерних)
  apps/api/AGENTS.md
    → Переопределяется только: "Этот сервис использует Python"
```

### Шаг 4: Начальная настройка для конкретных инструментов

Добавляет мосты к конфигурациям конкретных инструментов, чтобы **все AI-инструменты читали** сгенерированный AGENTS.md.

```
┌──────────────┐     ┌─────────────┐     ┌─────────────┐
│ Claude Code  │     │   Cursor    │     │   Codex     │
│  CLAUDE.md   │     │  .mdc rules │     │  AGENTS.md  │
│      ↓       │     │      ↓      │     │  (нативно)  │
│ "читать      │     │ "читать     │     │      ✓      │
│  AGENTS.md"  │     │  AGENTS.md" │     │             │
└──────┬───────┘     └──────┬──────┘     └─────────────┘
       └──────────┬─────────┘
                  ▼
           AGENTS.md (единый источник истины)
                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
  .ai-agents/  .ai-agents/  .ai-agents/
   context/     skills/      roles/
```

> **Принцип:** Файлы начальной настройки генерируются только для инструментов, которые уже используются. Конфигурационные файлы для неиспользуемых инструментов не создаются.

## Совместимость с инструментами

| Инструмент | Автоматически читает AGENTS.md | Начальная настройка |
|---|---|---|
| **OpenAI Codex** | Да (нативно) | Не требуется |
| **Claude Code** | Частично (запасной вариант) | Добавляет директиву в `CLAUDE.md` |
| **Cursor** | Нет | Добавляет `.mdc` в `.cursor/rules/` |
| **GitHub Copilot** | Нет | Генерирует `.github/copilot-instructions.md` |
| **Windsurf** | Нет | Добавляет директиву в `.windsurfrules` |
| **Aider** | Да | Добавляет чтение в `.aider.conf.yml` |

Автоматическая генерация начальных настроек:
```bash
bash scripts/sync-ai-rules.sh
```

## Иерархическая структура агентов

```
┌───────────────────────────────────────┐
│  Корневой PM-агент (AGENTS.md)        │
│  Глобальные соглашения + Правила      │
│  делегирования                        │
│  "Валидация дизайна > Валидация кода" │
└────────┬──────────┬─────────┬────────┘
         │          │         │
    ┌────▼────┐ ┌───▼────┐ ┌──▼─────┐
    │ Эксперт │ │ Инфра  │ │Планиро-│
    │ сервиса │ │  SRE   │ │вщик док│
    └─────────┘ └────────┘ └────────┘

Делегирование:  Родитель → Потомок (работает в рамках AGENTS.md этого каталога)
Отчётность:     Потомок → Родитель (сводка изменений после завершения задачи)
Координация:    Прямого взаимодействия между равными нет — косвенная координация через родителя
```

## Оптимизация токенов

| Формат | Количество токенов | Примечания |
|---|---|---|
| Описание API на естественном языке | ~200 токенов | |
| JSON DSL | ~70 токенов | **Экономия в 3 раза** |

**Пример api-spec.json:**
```json
{
  "service": "order-api",
  "apis": [{
    "method": "POST",
    "path": "/api/v1/orders",
    "domains": ["Order", "Payment"],
    "sideEffects": ["kafka:order-created", "db:orders.insert"]
  }]
}
```

**Целевой размер AGENTS.md:** менее **300 токенов** после подстановки

## Протокол восстановления сессии

```
Начало сессии:
  1. Прочитать AGENTS.md (большинство AI-инструментов делают это автоматически)
  2. Перейти по путям к файлам контекста и загрузить .ai-agents/context/
  3. Проверить .ai-agents/context/current-work.md (текущая работа)
  4. git log --oneline -10 (понять недавние изменения)

Конец сессии:
  1. Незавершённая работа → Записать в current-work.md
  2. Новые знания о домене → Обновить файлы контекста
  3. Невыполненные TODO → Записать явно
```

## Поддержание контекста

При изменении кода файлы `.ai-agents/context/` необходимо обновлять соответственно.

```
API добавлен/изменён/удалён       →  Обновить api-spec.json
Схема БД изменена                 →  Обновить data-model.md
Спецификация событий изменена     →  Обновить event-spec.json
Бизнес-политика изменена          →  Обновить domain-overview.md
Внешняя интеграция изменена       →  Обновить external-integration.md
Конфигурация инфраструктуры       →  Обновить infra-spec.md
  изменена
```

> Если не обновлять, следующая сессия будет **работать с устаревшим контекстом**.
