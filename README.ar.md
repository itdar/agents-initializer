# ai-initializer

**مولّد سياق المشروع التلقائي لأدوات البرمجة بالذكاء الاصطناعي**

> يفحص مجلد مشروعك ويولّد تلقائيًا
> ملف `AGENTS.md` + سياق المعرفة/المهارات/الأدوار حتى يتمكن وكلاء الذكاء الاصطناعي من البدء بالعمل فورًا.

```
أمر واحد → تحليل المشروع → توليد AGENTS.md → يعمل مع أي أداة ذكاء اصطناعي
```

---

## الاستخدام

> **ملاحظة حول استهلاك التوكنات** — أثناء الإعداد الأولي، يقوم نموذج متقدم بتحليل المشروع بالكامل وتوليد عدة ملفات (AGENTS.md، .ai-agents/context/، .ai-agents/skills/، .ai-agents/roles/). قد يستهلك هذا عشرات الآلاف من التوكنات حسب حجم المشروع. هذه تكلفة لمرة واحدة فقط؛ الجلسات اللاحقة تحمّل السياق المُعَدّ مسبقًا وتبدأ فورًا.

```bash
# 1. اجعل الذكاء الاصطناعي يقرأ HOW_TO_AGENTS.md وسيتولى الباقي

# الخيار أ: الإنجليزية (موصى به — استهلاك أقل للتوكنات، أداء مثالي للذكاء الاصطناعي)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "Read HOW_TO_AGENTS.md and generate AGENTS.md tailored to this project"

# الخيار ب: لغة المستخدم (موصى به إذا كنت تخطط لتعديل AGENTS.md يدويًا)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "HOW_TO_AGENTS.md를 읽고 이 프로젝트에 맞게 AGENTS.md를 생성하라"

# موصى به: --model claude-opus-4-6 (أو أحدث) للحصول على أفضل النتائج
# موصى به: --dangerously-skip-permissions للتنفيذ المستقل دون انقطاع

# 2. ابدأ العمل مع الوكلاء المولَّدين
./ai-agency.sh
```

## لماذا تحتاج هذا؟

أدوات البرمجة بالذكاء الاصطناعي **تتعلم المشروع من الصفر** في كل جلسة.

| المشكلة | النتيجة |
|---|---|
| لا يعرف اتفاقيات الفريق | تناقضات في أسلوب الكود |
| لا يعرف خريطة الـ API الكاملة | يستكشف قاعدة الكود بالكامل في كل مرة (تكلفة إضافية +20%) |
| لا يعرف الإجراءات المحظورة | عمليات خطرة مثل الوصول المباشر لقاعدة بيانات الإنتاج |
| لا يعرف تبعيات الخدمات | تأثيرات جانبية غير متوقعة |

**ai-initializer** يحل هذه المشكلة — ولّد مرة واحدة، وأي أداة ذكاء اصطناعي ستفهم مشروعك فورًا.

## كيف يعمل

### الخطوة 1: فحص المشروع وتصنيفه

يستكشف المجلدات حتى عمق 3 مستويات ويصنّف تلقائيًا بناءً على أنماط الملفات.

```
deployment.yaml + service.yaml  →  k8s-workload
values.yaml (Helm)              →  infra-component
package.json + *.tsx            →  frontend
go.mod                          →  backend-go
Dockerfile + CI config          →  cicd
...19 نوعًا يتم اكتشافها تلقائيًا
```

### الخطوة 2: توليد السياق

يولّد ملفات المعرفة في `.ai-agents/context/` من خلال **التحليل الفعلي للكود** بناءً على الأنواع المكتشفة.

```
تم اكتشاف خدمة خلفية
  → فحص المسارات/المتحكمات → توليد api-spec.json
  → فحص الكيانات/المخططات  → توليد data-model.md
  → فحص إعدادات Kafka      → توليد event-spec.json
```

### الخطوة 3: توليد AGENTS.md

يولّد ملف AGENTS.md لكل مجلد باستخدام القوالب المناسبة.

```
AGENTS.md الجذر (الاتفاقيات العامة)
  → الإيداعات: Conventional Commits
  → طلبات السحب: قالب مطلوب، موافقة واحدة على الأقل
  → الفروع: feature/{ticket}-{desc}
       │
       ▼ يُورَث تلقائيًا (لا يُكرَّر في الملفات الفرعية)
  apps/api/AGENTS.md
    → تجاوزات فقط: "هذه الخدمة تستخدم Python"
```

### الخطوة 4: التهيئة الخاصة بالأدوات

يضيف جسور إلى إعدادات الأدوات المختلفة حتى **تقرأ جميع أدوات الذكاء الاصطناعي** ملف AGENTS.md المولَّد.

```
┌──────────────┐     ┌─────────────┐     ┌─────────────┐
│ Claude Code  │     │   Cursor    │     │   Codex     │
│  CLAUDE.md   │     │  .mdc rules │     │  AGENTS.md  │
│      ↓       │     │      ↓      │     │  (أصلي)     │
│ "اقرأ        │     │ "اقرأ       │     │      ✓      │
│  AGENTS.md"  │     │  AGENTS.md" │     │             │
└──────┬───────┘     └──────┬──────┘     └─────────────┘
       └──────────┬─────────┘
                  ▼
           AGENTS.md (مصدر الحقيقة الوحيد)
                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
  .ai-agents/  .ai-agents/  .ai-agents/
   context/     skills/      roles/
```

> **المبدأ:** لا تُولَّد ملفات التهيئة إلا للأدوات المستخدمة فعلًا. لا يتم إنشاء ملفات إعدادات لأدوات غير مستخدمة أبدًا.


## هيكل الوكلاء الهرمي

```
┌───────────────────────────────────────┐
│  وكيل مدير المشروع الجذر (AGENTS.md) │
│  الاتفاقيات العامة + قواعد التفويض   │
│  "التحقق من التصميم > التحقق من الكود"│
└────────┬──────────┬─────────┬────────┘
         │          │         │
    ┌────▼────┐ ┌───▼────┐ ┌──▼─────┐
    │ خبير   │ │ البنية │ │ مخطط  │
    │ الخدمة │ │ التحتية│ │ التوثيق│
    └─────────┘ └────────┘ └────────┘

التفويض: الأب → الابن (يعمل ضمن نطاق AGENTS.md الخاص بذلك المجلد)
التقارير: الابن → الأب (ملخص التغييرات بعد إتمام المهمة)
التنسيق: لا تنسيق مباشر بين الأقران — تنسيق غير مباشر عبر الأب
```

## تحسين استهلاك التوكنات

| الصيغة | عدد التوكنات | ملاحظات |
|---|---|---|
| وصف الـ API بلغة طبيعية | ~200 توكن | |
| JSON DSL | ~70 توكن | **توفير 3 أضعاف** |

**مثال على api-spec.json:**
```json
{
  "service": "order-api",
  "apis": [{
    "method": "POST",
    "path": "/api/v1/orders",
    "domains": ["Order", "Payment"],
    "sideEffects": ["kafka:order-created", "db:orders.insert"]
  }]
}
```

**هدف AGENTS.md:** أقل من **300 توكن** بعد الاستبدال

## بروتوكول استعادة الجلسة

```
بدء الجلسة:
  1. قراءة AGENTS.md (معظم أدوات الذكاء الاصطناعي تفعل ذلك تلقائيًا)
  2. اتباع مسارات ملفات السياق لتحميل .ai-agents/context/
  3. التحقق من .ai-agents/context/current-work.md (العمل الجاري)
  4. git log --oneline -10 (فهم التغييرات الأخيرة)

نهاية الجلسة:
  1. العمل الجاري → تسجيله في current-work.md
  2. معرفة المجال المكتسبة حديثًا → تحديث ملفات السياق
  3. المهام غير المكتملة → تسجيلها بشكل صريح
```

## صيانة السياق

عند تغيير الكود، يجب تحديث ملفات `.ai-agents/context/` وفقًا لذلك.

```
إضافة/تعديل/حذف API          →  تحديث api-spec.json
تغيير مخطط قاعدة البيانات     →  تحديث data-model.md
تغيير مواصفات الأحداث         →  تحديث event-spec.json
تغيير سياسة العمل             →  تحديث domain-overview.md
تغيير التكامل الخارجي         →  تحديث external-integration.md
تغيير إعدادات البنية التحتية  →  تحديث infra-spec.md
```

> عدم التحديث يعني أن الجلسة التالية ستعمل **بسياق قديم**.
