# ai-initializer

**AI कोडिंग टूल्स के लिए स्वचालित प्रोजेक्ट संदर्भ जनरेटर**

> आपकी प्रोजेक्ट डायरेक्टरी को स्कैन करता है और स्वचालित रूप से
> `AGENTS.md` + ज्ञान/कौशल/भूमिका संदर्भ उत्पन्न करता है ताकि AI एजेंट तुरंत काम शुरू कर सकें।

```
एक कमांड → प्रोजेक्ट विश्लेषण → AGENTS.md जनरेशन → किसी भी AI टूल के साथ काम करता है
```

---

## उपयोग

> **टोकन उपयोग सूचना** — प्रारंभिक सेटअप के दौरान, एक शीर्ष-स्तरीय मॉडल पूरे प्रोजेक्ट का विश्लेषण करता है और कई फाइलें उत्पन्न करता है (AGENTS.md, .ai-agents/context/, .ai-agents/skills/, .ai-agents/roles/)। प्रोजेक्ट के आकार के आधार पर यह दसियों हज़ार टोकन खर्च कर सकता है। यह एक बार की लागत है; बाद के सत्र पूर्व-निर्मित संदर्भ लोड करते हैं और तुरंत शुरू होते हैं।

```bash
# 1. AI को HOW_TO_AGENTS.md पढ़ने दें और यह बाकी सब संभाल लेगा

# विकल्प A: अंग्रेज़ी (अनुशंसित — कम टोकन लागत, इष्टतम AI प्रदर्शन)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "Read HOW_TO_AGENTS.md and generate AGENTS.md tailored to this project"

# विकल्प B: उपयोगकर्ता की भाषा (अनुशंसित यदि आप AGENTS.md को मैन्युअल रूप से संपादित करने की योजना बनाते हैं)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "HOW_TO_AGENTS.md를 읽고 이 프로젝트에 맞게 AGENTS.md를 생성하라"

# अनुशंसित: --model claude-opus-4-6 (या बाद का) सर्वोत्तम परिणामों के लिए
# अनुशंसित: --dangerously-skip-permissions निर्बाध स्वायत्त निष्पादन के लिए

# 2. उत्पन्न एजेंट्स के साथ काम शुरू करें
./ai-agency.sh
```

## आपको इसकी आवश्यकता क्यों है?

AI कोडिंग टूल्स **हर सत्र में प्रोजेक्ट को शून्य से फिर से सीखते हैं**।

| समस्या | परिणाम |
|---|---|
| टीम की परिपाटियाँ नहीं जानता | कोड शैली में असंगतियाँ |
| पूर्ण API मानचित्र नहीं जानता | हर बार पूरे कोडबेस की खोज करता है (लागत +20%) |
| प्रतिबंधित कार्य नहीं जानता | जोखिमपूर्ण संचालन जैसे सीधे प्रोडक्शन DB एक्सेस |
| सेवा निर्भरताएँ नहीं जानता | छूटे हुए दुष्प्रभाव |

**ai-initializer** इसे हल करता है — एक बार उत्पन्न करें, और कोई भी AI टूल आपके प्रोजेक्ट को तुरंत समझ लेगा।

## यह कैसे काम करता है

### चरण 1: प्रोजेक्ट स्कैन और वर्गीकरण

गहराई 3 तक डायरेक्टरी की खोज करता है और फाइल पैटर्न के आधार पर स्वचालित वर्गीकरण करता है।

```
deployment.yaml + service.yaml  →  k8s-workload
values.yaml (Helm)              →  infra-component
package.json + *.tsx            →  frontend
go.mod                          →  backend-go
Dockerfile + CI config          →  cicd
...19 प्रकार स्वचालित रूप से पहचाने जाते हैं
```

### चरण 2: संदर्भ उत्पादन

पहचाने गए प्रकारों के आधार पर **वास्तव में कोड का विश्लेषण करके** `.ai-agents/context/` ज्ञान फाइलें उत्पन्न करता है।

```
बैकएंड सेवा पहचानी गई
  → रूट्स/कंट्रोलर्स स्कैन करें → api-spec.json उत्पन्न करें
  → एंटिटीज़/स्कीमा स्कैन करें  → data-model.md उत्पन्न करें
  → Kafka कॉन्फ़िग स्कैन करें    → event-spec.json उत्पन्न करें
```

### चरण 3: AGENTS.md उत्पादन

उचित टेम्पलेट्स का उपयोग करके प्रत्येक डायरेक्टरी के लिए AGENTS.md उत्पन्न करता है।

```
मूल AGENTS.md (वैश्विक परिपाटियाँ)
  → कमिट्स: Conventional Commits
  → PR: टेम्पलेट आवश्यक, 1+ अनुमोदन
  → शाखाएँ: feature/{ticket}-{desc}
       │
       ▼ स्वचालित रूप से विरासत में (चाइल्ड में दोहराया नहीं जाता)
  apps/api/AGENTS.md
    → केवल ओवरराइड: "यह सेवा Python का उपयोग करती है"
```

### चरण 4: वेंडर-विशिष्ट बूटस्ट्रैप

वेंडर-विशिष्ट कॉन्फ़िग में ब्रिज जोड़ता है ताकि **सभी AI टूल्स** उत्पन्न AGENTS.md को पढ़ सकें।

```
┌──────────────┐     ┌─────────────┐     ┌─────────────┐
│ Claude Code  │     │   Cursor    │     │   Codex     │
│  CLAUDE.md   │     │  .mdc rules │     │  AGENTS.md  │
│      ↓       │     │      ↓      │     │  (मूल)      │
│ "read        │     │ "read       │     │      ✓      │
│  AGENTS.md"  │     │  AGENTS.md" │     │             │
└──────┬───────┘     └──────┬──────┘     └─────────────┘
       └──────────┬─────────┘
                  ▼
           AGENTS.md (सत्य का एकल स्रोत)
                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
  .ai-agents/  .ai-agents/  .ai-agents/
   context/     skills/      roles/
```

> **सिद्धांत:** बूटस्ट्रैप फाइलें केवल उन वेंडर्स के लिए उत्पन्न होती हैं जो पहले से उपयोग में हैं। अप्रयुक्त टूल्स के लिए कॉन्फ़िग फाइलें कभी नहीं बनाई जातीं।

```json
{
  "service": "order-api",
  "apis": [{
    "method": "POST",
    "path": "/api/v1/orders",

    "sideEffects": ["kafka:order-created", "db:orders.insert"]
  }]
}
```

## संदर्भ रखरखाव

जब कोड बदलता है, तो `.ai-agents/context/` फाइलों को तदनुसार अपडेट किया जाना चाहिए।

```
API जोड़ी/बदली/हटाई गई        →  api-spec.json अपडेट करें
DB स्कीमा बदली                 →  data-model.md अपडेट करें
इवेंट विनिर्देश बदला            →  event-spec.json अपडेट करें
व्यावसायिक नीति बदली            →  domain-overview.md अपडेट करें
बाहरी एकीकरण बदला             →  external-integration.md अपडेट करें
इन्फ्रास्ट्रक्चर कॉन्फ़िग बदली   →  infra-spec.md अपडेट करें
```

> अपडेट न करने का अर्थ है कि अगला सत्र **पुराने संदर्भ के साथ काम करेगा**।
